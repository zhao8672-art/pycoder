核心发现: PyCoder 在 Clean Architecture 上「有皮无骨」——定义了 3 个核心 Protocol 和 3 个 Adapter，但 server 层 30+ 个文件仍然直接依赖 ChatBridge。端口是装饰品，不是基础设施。

4. 沙箱与安全 — 多层防御缺口

层级PyCoder 现状行业标准差距
L1: 代码执行 子进程 subprocess (Windows不安全) Docker/gVisor/Firecracker 缺少容器隔离
L2: 文件访问 safe_path() 工作区限制 chroot/AppArmor/SELinux 用户空间检查，内核无防护
L3: 命令执行 白名单包含 rm/del/mv 最小权限原则 允许破坏性命令
L4: 权限策略 客户端检查，Agent可绕过 服务端强制 ACL 无服务端 enforcement
L5: 网络隔离 无 网络策略/egress控制 完全缺失
最大风险: 自主流水线模式下，ALLOWED_COMMANDS 包含 rm、del、mv。如果 Agent 被对抗性提示词攻击，可以直接删除项目文件，权限系统无法阻止。

6. 更好的架构方案 — 三项可选升级路径

方案A: 微内核 + 插件架构 推荐

┌────────────────────────────────────────────────────┐ │ PyCoder 微内核 │ │ ┌──────────┐ ┌──────────┐ ┌──────────────────┐ │ │ │ EventBus │ │ Registry │ │ DependencyInject │ │ │ │(异步消息) │ │(服务发现) │ │ (DI Container) │ │ │ └──────────┘ └──────────┘ └──────────────────┘ │ ├────────────────────────────────────────────────────┤ │ 插件 (通过 Protocol 注册) │ │ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌────────┐ │ │ │Agent │ │Model │ │Sandbox│ │RepoMap│ │Skills │ │ │ │Engine│ │Router│ │ Mgmt │ │(PageR)│ │Market │ │ │ └──────┘ └──────┘ └──────┘ └──────┘ └────────┘ │ └────────────────────────────────────────────────────┘

优势: 每个子系统可独立开发/测试/替换。新模型接入只需实现 LLMProvider Protocol 并注册。 Aider 的实际架构就是这种模式——所有组件通过 Coder 类组合，而非硬 import。

方案B: Plan-Act 双模式 Cline风格

用户输入 → ModeRouter → ┌─ PlanMode: read-only, 收集上下文, 制定计划 └─ ActMode: tool-use loop, 每步审批

Cline 的 Plan/Act 分离是其最大的架构优势。PyCoder 可以在 UnifiedAgent 中增加 ModeRouter， 在 plan 阶段禁止写操作，仅允许 read_file/search_files。

方案C: Memory Bank Cline风格

.pycoder/memory/ ├── project_brief.md ← 项目概述（自动生成，人工审阅） ├── product_context.md ← 产品需求上下文 ├── active_context.md ← 当前会话上下文 ├── system_patterns.md ← 架构决策记录 ├── tech_context.md ← 技术栈和环境 └── progress.md ← 进度追踪

替代当前简单的 SQLite 会话存储，提供跨会话、跨重启的持久记忆。 Aider 通过 .aider 目录和 Git history 实现类似功能。

7. 升级路线图 — 分阶段重构计划

P0
v0.6
依赖注入基础设施 + 端口反向注入
引入轻量 DI（如 python-dependency-injector 或手写 Registry）。 autonomous_pipeline、task_decomposer 改为接受 LLMProvider 而非 ChatBridge。 subprocess_sandbox 消除对 server 层的反向依赖。
效果: Clean Architecture 真实度 6.0 → 8.5
P0
v0.6
RepoMap — 代码仓库智能上下文映射
引入 tree-sitter 解析符号（借鉴 Aider 的 repomap.py），构建文件依赖图，用 PageRank 排序。 替代当前的滑窗截断，使 Agent 在大型项目（100+ 文件）中能精准理解上下文。
效果: 上下文管理 4.5 → 8.0，大项目可用性从不可用到可用
P1
v0.7
Plan/Act 双模式分离
在 UnifiedAgent 中增加 ModeRouter。Plan 模式下仅允许 read_file/search_files/grep， 禁止写操作。输出结构化计划供用户审批后再进入 Act 模式。
效果: 安全性和用户体验质的提升
P1
v0.7
Memory Bank — 跨会话持久记忆
实现 .pycoder/memory/ 文件系统，自动维护 project_brief/product_context/progress。 每次会话启动时注入相关 memory 到 system prompt。
效果: 从无状态助手变为有记忆的开发伙伴
P1
v0.7
模型路由智能化 + 多供应商开放
TaskClassifier 增加 LLM 二次分类层。ModelCapabilityMatrix 增加运行时性能反馈。 增加 Claude/Gemini/Llama 适配路径（通过 LiteLLM 或直接适配）。
效果: 模型路由 3.5 → 7.0，解除 DeepSeek 锁定
P1
v0.7
权限系统服务端化
将权限检查从纯客户端移到服务端。引入 ACL 规则引擎（allow/ask/deny + per-path + per-operation）。 自主模式下自动降权（禁止 rm/del）。
效果: 安全性 7.0 → 9.0
P2
v0.8
Docker 沙箱默认启用 + 网络隔离
docker_sandbox.py 作为默认沙箱实现。增加 egress 网络策略控制。 用户代码执行完全隔离于宿主机。
效果: 沙箱安全对标 Modal/Fly.io
P2
v0.8
Native Tool Use 协议（OpenAI/Anthropic 兼容）
在 ReAct JSON 格式之外增加 OpenAI function calling / Anthropic tool_use 原生支持。 让支持 tool_use 的模型使用原生能力而非自定义 JSON 解析。
效果: 工具调用成功率 70% → 95%（对支持原生 tool_use 的模型）
P3
v0.9
数据库 Schema 迁移 + SQLAlchemy 统一
引入 Alembic 进行 schema 版本管理。统一所有持久化到 SQLAlchemy ORM。 metadata 字段改为结构化 Pydantic model。
效果: 数据层 5.0 → 8.0
P3
v0.9
向量化上下文检索 + RAG
对项目文件做 embedding（ChromaDB/lanceDB），Agent 可以语义检索相关代码片段。 替代当前的全文搜索和滑窗截断。Cursor 级别的上下文理解能力。
效果: 上下文管理达到行业领先水平

8. 总结 — PyCoder 在竞争格局中的位置

PyCoder 的三大护城河（应加倍投入）:
  1. 动态反思频率调整 — 竞品中独一无二，这是真正的算法创新
  2. 7步自主流水线 — 从需求到交付的完整闭环，Aider/Cline 都没有
  3. Skills 市场 + 多源聚合 — 生态化思维，MCP 集成已领先
必须立刻修复的架构债务（阻止规模化）:
  1. 端口空洞化 — Clean Architecture 是假的，不改则架构永远混乱
  2. 缺少 RepoMap — 大项目 (>50 files) 基本不可用
  3. 客户端权限检查 — 安全漏洞，尤其自主模式下的 rm/del 白名单
  4. DeepSeek 锁定 — 单点故障风险，且错过 Claude/Gemini 的能力
一句话定位: PyCoder 是「概念领先但执行落后」的选手——多 Agent 协作和自主流水线的设计视野超过 Aider 和 Cline，但 Clean Architecture 的空洞化、RepoMap 的缺失、模型生态的封闭使其在工程成熟度上大幅落后。修复 P0 项后，可进入第一梯队。
